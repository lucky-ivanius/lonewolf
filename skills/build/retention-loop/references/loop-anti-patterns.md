# Loop anti-patterns

Common ways users fake a retention loop. Reject these and force the real answer.

## "They will use it because it is useful"

Not a loop. Useful is necessary, not sufficient. The loop is what makes them open the app on day 2 specifically, not in theory.

Force the user to name the trigger that brings them back tomorrow.

## "We will send notifications"

Notifications are a delivery mechanism, not a loop. The question is what is in the notification that the user wants to see.

Force the user to name the content of the notification, not the channel.

## "They will tell their friends"

Word-of-mouth is acquisition, not retention. The user telling friends does not make THIS user come back. (It might make them more sticky as community grows, but that is a different loop, the social one, and needs to be articulated as such.)

Force the user to separate "what makes ME come back" from "what makes me invite others".

## "Daily price updates"

If the product is a price feed, this might be a real loop. If the product is anything else and the loop reduces to "users will check the price", the loop is too thin. Crypto is full of price feeds; one more is not differentiated.

Force the user to explain what the price update enables them to DO, not just see.

## "Ranking and leaderboards"

Leaderboards work as a loop when the user has a stake in their position. If the leaderboard is decoration with no consequence, it is theater.

Force the user to articulate what changes about the user's experience when they move up or down the leaderboard.

## "Daily quests"

Quests are a loop if they generate real reward AND the reward decays without daily action. If the quests are detached from real product value, they are slot-machine theater.

Force the user to tie quest reward to product value.

## "Compounding state"

Some products are stickier the longer the user uses them (more history, more saved settings, more network connections). This is a real moat IF the user feels it. If the compounding state is invisible to the user, it does not retain.

Force the user to name what the user SEES that gets better over time.

## "Generic streaks"

Duolingo-style streaks work because language learning has a clear daily action and a clear loss when the streak breaks. Streaks bolted onto a product without daily-action fit are theater.

Force the user to confirm there is a real daily action that streaks can attach to.

## "Tokens / rewards / airdrops"

Token rewards retain only as long as the token has speculative value. The day the token loses value, the loop breaks. Use as a temporary boost, not a real loop.

Force the user to articulate the loop that survives token price going to zero.

## "We will A/B test until something works"

This is a process, not a loop. Force the user to name a hypothesis specific enough to A/B test, then test it. Without a hypothesis, A/B testing produces random noise.

## How to push back

When the user offers an anti-pattern:

1. Name it specifically: "That sounds like the 'they will use it because it is useful' anti-pattern."
2. Restate the real question: "What specifically brings them back on day 2?"
3. If they cannot answer, mark the anchor as `unvalidated` rather than accept theater.

A `partial` or `no` verdict is a useful output. A `yes` verdict from theater is harmful slop.
