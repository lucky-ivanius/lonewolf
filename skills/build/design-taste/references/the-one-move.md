# The one move

By page type, the single highest-impact change. Apply this first; everything else is incremental.

## Landing page

**Move**: Replace the gradient hero + headline + CTA with one specific asset (a screenshot of the actual product, ideally captured at the moment of value). Pair with a one-sentence headline that names the user's specific problem.

The hero is what 90% of visitors see. A specific hero beats a polished generic hero every time.

## Dashboard

**Move**: Lead with the single number the user came to see. Surface it big, in tabular figures, with the secondary context below.

Most dashboards bury the lead under navigation, tabs, and chrome. The user wants the number; give them the number first.

## Wallet / address page

**Move**: Show the wallet's identity (name-service handle if available, or ellipsized address) and current native-token balance as the largest typographic element. Everything else descends from there.

A wallet page is for confirming "this is my wallet, this is my balance". Lead with both.

## Transaction confirmation flow

**Move**: At the moment the user clicks "confirm", show only what they need to see: amount, destination, gas. Hide everything else behind a "details" disclosure.

Most confirmation surfaces dump every internal call, every raw input, every fee detail. That is for power users in the explorer; in the app, it adds friction.

## NFT marketplace listing

**Move**: Make the image take 60-70% of the above-the-fold space. Title and seller below, price prominent on the right, "Buy" button at the same prominence as the price.

Marketplaces that bury the image and elevate metadata read like spreadsheets, not stores.

## Documentation page

**Move**: Replace the table of contents with a one-paragraph orientation: what this page is, who it is for, what they will be able to do after reading. Then the actual content.

Docs pages that lead with a giant TOC make the reader feel they have to read everything before knowing if they are in the right place.

## Pricing page

**Move**: Lead with the answer the user came for: the cheapest workable tier with its price. Then the comparison table.

Pricing pages that lead with feature comparison hide the answer.

## Settings page

**Move**: Group settings by frequency of use, not by feature area. Top group: things people change weekly. Bottom group: things people change once.

Default settings ordering (alphabetic, by feature, by team that built it) makes users hunt.

## Empty state

**Move**: Replace "Nothing here yet" with one sentence that names the action that creates the first item, then the button to do that action.

Empty states are wasted real estate by default. They are the moment the user is most uncertain and most willing to act.

## Error state

**Move**: Lead with what the user can do next, not what failed. "Try again" or "Contact support" as the first thing they read, then the failure detail in smaller text below for context.

Error pages that lead with the failure and bury the recovery train users to give up.

## How to use this list

When activated as part of `design-taste`, the skill picks the page type the user is asking about, applies the move at the top, and only then walks the wider markers checklist. The one move first ensures the highest-impact change happens, even if the user runs out of time before the rest.
